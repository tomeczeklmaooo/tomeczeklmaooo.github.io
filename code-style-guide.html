Code Style Guide

Introduction

This page defines code style guidelines for my projects (some apply to the C programming language only).

1. General

* Code must target the C23 standard, but compile without warnings on compilers that support C99 or later.

* All source and header files must use UTF-8 encoding, LF line endings, and feature a trailing newline.

* A single line of code cannot exceed 100 characters in length.

* Use tabs for indentation (preferred width: 4), spaces for alignment only.

2. Naming

* Function and variable names should use lower snake case.

* Pointer and reference symbols belong next to the variable name, not the type. 

int *pointer = NULL;
int *reference = &something;

* Constants and macros should use upper snake case.

* Structs and enums should use lower snake case (although pascal case is acceptable as well).

* Filenames should use lower snake case.

* Function and variable names should describe what the function/variable is, without Hungarian notation (or whatever similar).

3. Declaration and initialization of variables

* Declare one variable per line, this helps with debugging and readability.

* Initialize as close to first use as possible.

* Prefer using const to #define when possible.

* Avoid variables with global scope.

* Keep the amount of local variables (eg. in a function) below 10.

4. Functions

* A function should perform a single task.

* Keep functions below 60 lines in length (excluding braces and comments).

* Make use of early returns if errors were encountered.

* Use modern prototypes with full parameter types.

* If there is something that can go wrong in a function, make it return an integer.

5. Includes and headers

* All includes should appear at the very top of the source file.

* Includes should follow the order: standard headers, external library headers, project headers.

* Every header must have include guards. 

#ifndef HEADER_H
#define HEADER_H
... definitions go here ...
#endif

* Header files should expose only functions and data structures that might be required by other source files.

6. Error handling

* Never ignore return values.

* Log errors to stderr, use fprintf(stderr, ...) for that.

* Exit with nonzero codes on fatal failures.

* Avoid using exit() in places where we do not want to end the entire process.

7. Formatting

* Opening and closing braces should be on their own lines, nothing else other than comments can be on the same line.
An exception to this rule is when the character directly before the opening brace is =: 

char *array[] = {
	...
}

* Use braces at all times, for single lines of code in an if (...) as well.

* Function declarations should have a leading blank line.

* Avoid putting too many blank lines.

8. Comments

* Use // for short comments, /* ... */ for longer comments.

* Use comments to explain why your code does the task, what you code does should be visible from the code itself.

* Document public functions with their purpose, parameters and return values.

9. Build and tools

* Use clang to compile.

* Use the included Makefile to make your life easier (and to be consistent).

* Compiler should not emit any warnings or errors.

10. Version control

* Commit messages should be descriptive, not vague like "do stuff".
